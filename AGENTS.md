## 4. Соглашения по коду

- **TypeScript strict.** Никаких `any` без крайней нужды; для неизвестных ответов API — `unknown` + сужение типов (как уже сделано в `apiService.ts`).
- **Все обращения к backend — только через `apiService`.** Не вызывать `axios` напрямую из компонентов. Новые эндпоинты добавлять методом в класс `ApiService`, типы ответов — рядом (`export type ...Response`).
- **Стили:** использовать переменные из `styles/tokens.css`, не хардкодить цвета/отступы. У каждого компонента — свой `.css` рядом с `.tsx`.
- **Именование:** компоненты — `PascalCase`, файл компонента совпадает с именем; хуки — `useCamelCase`.
- **Ассеты** из Figma класть в `assets/<экран>/` и импортировать как модуль.
- Перед коммитом крупных изменений убедиться, что `npm run build` и `npm run lint` проходят.

## 5. Git: совместная работа двух агентов

Над репозиторием одновременно работают два ИИ-агента. Чтобы не ломать работу друг друга:

### 5.1 Формат сообщения коммита

```
<type>(<scope>): <краткое описание> [<author>]

<тело: что и зачем изменено>

Touched: <ключевые файлы/папки>
Note: <сообщение для второго агента — если нужно>
Next: <что логично сделать дальше / что оставлено недоделанным>
```

- **type:** `feat` | `fix` | `refactor` | `style` | `docs` | `chore` | `wip`
- **scope:** область, напр. `public-home`, `login`, `dashboard`, `api`, `ui-kit`, `layout`, `chat`, `types`
- **author:** кто сделал коммит — `[claude]` или `[ai-2]` (замени на реальное имя второго агента). Люди тег не ставят.

**Пример:**
```
feat(dashboard): добавлен блок последних поездок [claude]

Выведен список из getTrips(), скелетоны на время загрузки.
Touched: src/pages/Dashboard.tsx, src/api/apiService.ts
Note: ai-2 — типизацию TripItem вынес в types/index.ts, переиспользуй.
Next: верстка пустого состояния (нет поездок) по макету Figma.
```

### 5.2 Правила, снижающие конфликты

1. **Маленькие частые коммиты.** Не копи десятки изменённых файлов в один коммит.
2. **Один агент — одна область за раз.** Перед началом работы укажи область в секции «Активные работы» ниже (см. 5.3) и закоммить это изменение AGENTS.md первым. Так второй агент видит, что зона занята.
3. **`git pull` перед началом и перед каждым коммитом.** Работаем по rebase: `git pull --rebase`.
4. **Не переформатируй файлы, которые не меняешь по сути.** Массовый автоформат создаёт ложные конфликты.
5. **`apiService.ts` меняем только аддитивно** (добавляем методы/типы), не переписываем существующие сигнатуры без пометки `Note:` для второго агента.
6. **Общие файлы-«узкие места»** (`App.tsx`, `apiService.ts`, `types/`, `styles/tokens.css`) трогаем осторожно и маленькими правками.
7. Если видишь незакоммиченные чужие изменения — **не затирай**, оставь и сообщи через `Note:` в своём следующем коммите.
