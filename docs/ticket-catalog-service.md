# Ticket Catalog Service

Краткий рабочий конспект по `ticket-catalog-service`, чтобы не держать контекст только в загрузках.

## Назначение

- Сервис получает ответ от ЛКП API.
- Обогащает его каталогом продуктов из БД.
- Отдаёт фронту готовую структуру экранов покупки билетов и пополнения.
- Использует кэш Redis и перевод строк через отдельный translations service.

## Ключевые клиентские ручки

- `GET /v1/tickets/cards/{linkedCardId}/products`
  - для авторизованной / привязанной карты;
  - `linkedCardId` приходит из linked cards.

- `GET /v1/tickets/cards/by-uid/{cardUid}/products`
  - для неавторизованного сценария;
  - сначала вводится номер транспортной карты;
  - затем `cardUid` добывается через `GET /cards/v1.0/{cardNumber}/validate`;
  - после этого запрашиваются продукты по `cardUid`.

## Что приходит в ответе

- `card`
  - номер / uid / тип карты;
  - для виртуальных карт и отдельных сценариев могут приходить дополнительные поля.
- `availableProducts`
  - `wallet` с `priceMin`, `priceMax`, `paymentTypes`;
  - категории `UNIFIED`, `TAT`, `TRAIN`;
  - секции `durationList`, `betweenZones`;
  - `products[]`, `options[]`, `badges[]`.

## Важные правила для фронта

- Для гостя не показывать привязанную банковскую карту среди способов оплаты.
- Если `sbp.state === NONE`, SBP не отображать.
- Категории и иконки лучше строить на уровне UI-компонентов, а не хранить как разрозненные картинки.
- Для билетов важны `isRecommended`, `isFreezable`, `paymentTypes`, `priceDelta`, `durationDays`.
- Для wallet-пополнения ориентироваться на `priceMin` / `priceMax`.

## Что ещё полезно помнить

- Сервис использует кэш с TTL около 60 секунд.
- Переводы подставляются по ключам вида `products_{productId}_{attr}` и `badges_{badgeId}_{attr}`.
- Админка ticket products отдельно управляет каталогом, бейджами, иконками и пробелами каталога.
